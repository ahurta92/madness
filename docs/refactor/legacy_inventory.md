# Legacy `molresponse` Inventory

## Scope

- Directory scanned: `src/apps/molresponse_legacy/`
- There are no subdirectories with additional code.
- This tree is a frozen compatibility import, not a full historical re-host.
- Built sources per `CMakeLists.txt`: `TDDFT.cc`, `density.cc`, `property.cc`, `global_functions.cc`, `basic_operators.cc`, `calc_runner.cc`, `timer.cc`, `Plot_VTK.cc`, `iter_freq2.cc`, `iterate_excited.cc`, `iterate_gamma.cc`, `Lambda_X.cc`, `Theta_X.cc`, `molresponse.cc`.
- Explicitly excluded from the build: `property_functions.cc`, `iterate_freq.cc`, `iterate_xy.cc`.

## Top-Level Algorithmic Flow

### Main executable path

1. Start MADNESS, parse the input filename, and read response plus ground-state archives.
   - Implemented by `main` in `molresponse.cc:76`, `read_and_create_density` in `molresponse.cc:57`, `ResponseParameters::read_and_set_derived_values` in `response_parameters.h:298`, and `GroundParameters::read` in `ground_parameters.h:54`.
2. Construct a `density_vector` matching the requested calculation mode, then construct `TDDFT`.
   - Implemented by `set_density_type` in `density.cc:290` and `TDDFT::TDDFT` in `TDDFT.cc` (constructor declaration at `TDDFT.h:190`).
3. Build a coarse protocol/load-balance setup if running with more than one rank.
   - Implemented by `TDDFT::set_protocol` in `calc_runner.cc:68`, `TDDFT::make_nuclear_potential` in `TDDFT.cc:852`, and `TDDFT::initial_load_bal` in `TDDFT.cc:231`.
4. Select the first protocol threshold and dispatch to either excited-state or first-order response.
   - Implemented by `main` in `molresponse.cc:76`.
5. For each protocol threshold:
   - update MADNESS numerical defaults,
   - reproject archived/working functions to the current `k` and threshold,
   - either restart from archive or create an initial guess,
   - iterate until density and BSH residual tests pass or iteration limit is hit.
   - Excited-state driver: `TDDFT::solve_excited_states` in `calc_runner.cc:558`, `TDDFT::iterate_excited` in `iterate_excited.cc:29`.
   - Frequency-response driver: `TDDFT::solve_response_states` in `calc_runner.cc:750`, `TDDFT::iterate_freq2` in `iter_freq2.cc:27`.
6. Optionally save restart files and plot orbitals/densities.
   - Implemented by `TDDFT::save` in `TDDFT.cc:159`, `do_vtk_plots` in `Plot_VTK.cc:25` and `Plot_VTK.cc:117`, and plotting helpers in `TDDFT.cc` near the bottom of the file.
7. For dipole response, compute and print the polarizability tensor.
   - Implemented by `TDDFT::polarizability` in `TDDFT.cc:4014` and `TDDFT::PrintPolarizabilityAnalysis` in `TDDFT.cc:4013`.

### Excited-state iteration flow

1. Build projector, XC operator, KAIN solvers, residual workspaces, and convergence thresholds.
   - `TDDFT::iterate_excited` in `iterate_excited.cc:29`
2. At each iteration, recompute transition densities and optionally redistribute work.
   - `TDDFT::make_density` in `TDDFT.cc:2016`, `TDDFT::loadbal` in `TDDFT.cc:240`
3. Form residual diagnostics from density changes and BSH residuals.
   - `TDDFT::iterate_excited` in `iterate_excited.cc:29`
4. Update the excited-state subspace and energies.
   - `TDDFT::update_x_space_excited` declared in `TDDFT.h:411`, implemented in `TDDFT.cc`
5. Sort final roots, print diagnostics, and run transition-property analysis.
   - `TDDFT::sort` in `TDDFT.cc:3053`, `TDDFT::analysis` in `iterate_excited.cc:300`, `TDDFT::analyze_vectors` in `TDDFT.cc:4264`

### Frequency-response iteration flow

1. Build the property RHS (`PQ`) for dipole or nuclear perturbations.
   - `TDDFT::PropertyRHS` in `TDDFT.cc:730`, called from `TDDFT::solve_response_states` in `calc_runner.cc:750`
2. Build response BSH operators for `+omega` and `-omega`, including energy shifts when needed.
   - `TDDFT::make_bsh_operators_response` declared in `TDDFT.h:354`, implemented in `TDDFT.cc`
3. At each iteration, recompute transition density, load balance, and test density plus BSH residuals.
   - `TDDFT::iterate_freq2` in `iter_freq2.cc:27`
4. Apply the response update step with optional KAIN acceleration.
   - `TDDFT::update_x_space_response` declared in `TDDFT.h:371`, implemented in `TDDFT.cc`
5. Save and/or plot the converged response orbitals and densities.
   - `TDDFT::save` in `TDDFT.cc:159`, `do_vtk_plots` in `Plot_VTK.cc`

## Missing or Partial Features

- Saved-state naming convention:
  - Present only as user-provided filenames: `restart_file`, `save_file`, `save_density_file`, `load_density_file`.
  - There is no canonical protocol-aware or frequency-aware naming scheme generated by the code.
- Restart logic:
  - Present, but simple.
  - `TDDFT::save`/`TDDFT::load` serialize only `archive`, `tda`, dimensions, `omega`, and `Chi`.
  - `density_vector::SaveDensity`/`LoadDensity` serialize density-side objects separately.
  - No layered restart precedence exists; the code either loads the exact file the user names or starts fresh.
- Protocol handling / threshold ramping:
  - Present as a simple loop over `r_params.protocol()`.
  - `TDDFT::set_protocol` changes MADNESS defaults and `TDDFT::check_k` reprojects existing state.
  - There is no separate protocol metadata object, no per-point protocol history, and no protocol-specific snapshot hierarchy.
- Parallelization logic:
  - Present only as whole-world MADNESS/MPI execution with periodic load balancing.
  - `TDDFT::initial_load_bal`, `TDDFT::loadbal`, `TDDFT::orbital_load_balance`, and `LoadBalanceDeux` redistribute trees.
  - There is no subgroup/subworld scheduler, no point-parallel distribution, and no explicit task ownership metadata.
- Incomplete historical functionality:
  - `iterate_freq.cc`, `iterate_xy.cc`, and `property_functions.cc` are excluded because upstream dependencies are missing.
  - This means the checked-in tree is not a complete old-response implementation.

## Structure and Code-Quality Notes

- The code is very explicit about numerical workflow order; the driver functions are readable once the container types are understood.
- `response_space` and `X_space` are useful, focused containers that make vector-space operations concise.
- `ResponseParameters` and `GroundParameters` are thin compatibility wrappers rather than deeply entangled parser code.
- The weakest structural point is `TDDFT`: it is a very large god-object that mixes setup, operators, iteration, I/O, plotting, diagnostics, and property analysis.
- The legacy compatibility patches are called out clearly with `LEGACY_PATCH`, which makes local deviations easier to audit.

- The active solver/integration paths above were read directly from the code.
